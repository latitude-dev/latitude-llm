# Deployment

Deployment is single-branch by default. `development` is the trunk branch and normal merges to it deploy the merged commit to staging.

Production deployments are release-tag based. A production release starts as a reviewed PR that updates the root `CHANGELOG.md` with a monorepo release entry. Use `scripts/release.sh [version]` to open that PR. If no version is supplied, the script finds the latest merged `vX.Y.Z` tag and bumps to the next minor version, for example `v1.4.2` → `v1.5.0`; if no release tag exists, it starts at `v0.1.0`.

After the release PR merges into `development`, the normal staging deployment runs for that changelog commit. Once staging succeeds, the deploy workflow creates the monorepo release tag from the latest changelog heading, for example `v1.5.0`, pointing at the merged changelog commit. The tag push is the only production deployment trigger. The production run validates that the tag is a `vX.Y.Z` tag, that the tagged commit is reachable from `origin/development`, that the latest root changelog heading matches the tag, and that the tagged commit already has a successful staging deployment.

Public package publishing is decoupled from production deployment. Package publish workflows run on every merge to `development`; each package no-ops unless its package version differs from the registry. When a package version changes, publishing requires the package's own changelog entry for that version. Package versions and package changelogs remain developer-managed and are not generated by the monorepo release script.

Do not use `main` as a production promotion branch.
