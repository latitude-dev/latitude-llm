# MCP server

Latitude exposes an [MCP](https://modelcontextprotocol.io/) server so AI agents (Claude Code, Cursor, Codex, custom CLIs) can use Latitude's machine-facing capabilities as typed tools instead of curling raw HTTP. The server is mounted at `/v1/mcp` **inside the existing `apps/api` process** — there is no separate MCP service.

## Tools are derived from API routes

Every route declared via `defineApiEndpoint(...)` is registered as an MCP tool by default. Tool name = route `name` (camelCase), tool description = route `description`, tool input schema = the flattened union of `route.request.{params, query, body}`, tool output schema = the 2xx-JSON response schema.

Opt out per-route by passing `tool: false`. Internal lifecycle endpoints and web-only callbacks should opt out; everything else stays on.

A snapshot of the tool surface lives in `apps/api/mcp.json` and is regenerated by `pnpm mcp:emit`. CI guards drift.

For the full pipeline (HTTP / OpenAPI / MCP / SDK fan-out), see [`api.md`](./api.md).

## Per-request server, full middleware re-entry

The MCP route handler creates a fresh `McpServer` instance per request and dispatches each tool call by re-entering the Hono router via `app.fetch(internalRequest)`:

```
POST /v1/mcp → handler:
  mcpServer = new McpServer(...)
  for each registered tool descriptor:
    mcpServer.registerTool(name, schema, async (input) => {
      url = buildInternalUrl(prefix, pathTemplate, params, query)
      response = await app.fetch(new Request(`${BASE}${url}`, {
        method, headers: { Authorization, X-Forwarded-For, Content-Type },
        body: descriptor declares body ? JSON.stringify(body) : undefined,
      }))
      return { content: [...], structuredContent: JSON.parse(text), isError }
    })
  transport.handleRequest(c.req.raw)
```

Two consequences worth internalising:

1. **The full middleware chain re-runs on every tool call** — auth, rate-limit, org-context, validation. A tool call is not an escape hatch around any HTTP-level guard. Forward `Authorization` and `X-Forwarded-For` from the outer request so the inner validators see the same identity and IP.
2. **Tools dispatch by the *declared* body, not by HTTP method.** `GET` / `HEAD` routes never declare bodies in `createRoute`, so they're naturally bodyless. `DELETE` with a declared body sends it — useful for bulk-delete operations that need a selection in the request payload.

## OAuth2 cross-origin layout

The MCP server is the protected resource; the web app is the authorization server. They're on different origins, and the browser only ever talks to the web origin during sign-in/consent.

```
┌──────────────────────────────────────┐      ┌────────────────────────────────────┐
│ apps/web — app.latitude.so           │      │ apps/api — api.latitude.so         │
│ (TanStack Start)                     │      │ (Hono / OpenAPIHono)               │
│                                      │      │                                    │
│  /login              sign-in UI      │      │  /v1/...   REST + ApiKey/OAuth     │
│  /welcome            org-picker      │      │  /v1/mcp   MCP transport           │
│  /auth/consent       OAuth consent   │      │                                    │
│  /api/auth/*         BA handler      │      │  /.well-known/oauth-protected-     │
│    (incl. mcp/authorize, mcp/token,  │      │     resource                       │
```

### Discovery flow (RFC 9728 + RFC 8414)

1. MCP client → `api.latitude.so/v1/mcp` with no token → `401` + `WWW-Authenticate: Bearer resource_metadata="https://api.latitude.so/.well-known/oauth-protected-resource"`.
2. MCP client → `api.latitude.so/.well-known/oauth-protected-resource` → `{ "resource": "https://api.latitude.so", "authorization_servers": ["https://app.latitude.so/api/auth"] }`.
3. MCP client → `app.latitude.so/api/auth/.well-known/oauth-authorization-server` → Better Auth's AS metadata listing the authorize/token/register endpoints (all on the web origin).
4. MCP client opens the browser at `app.latitude.so/api/auth/mcp/authorize?…`. Same origin as the existing session cookie — Better Auth reads it directly. If no session, BA redirects to `/login`, user signs in, BA continues the authorize step.
5. BA mints a `consent_code` and redirects to `app.latitude.so/auth/consent?consent_code=…&client_id=…`. Still same origin.
6. User picks an org and approves. The web server fn (a) validates membership, (b) writes the chosen org id to `oauth_applications.organization_id` for the matching `client_id`, (c) calls `betterAuth.api.oauth2Consent({ accept: true, consent_code })` in-process.
7. BA returns the auth code to the MCP client's redirect URI.
8. MCP client → `POST app.latitude.so/api/auth/mcp/token` → opaque access token.
9. MCP client → `api.latitude.so/v1/mcp` with `Authorization: Bearer …`. The API's middleware validates the token (Drizzle SELECT, joins to `oauth_applications.organization_id`) and sets `c.var.auth` / `c.var.organization`. Tool dispatch proceeds.

The browser never crosses origins — the session cookie stays on the web app. The MCP client (a CLI agent) crosses origins for the token call and the protected-resource call, but those are header-bearer requests, not cookie requests, so CORS and `SameSite` don't apply.

### Org binding lives on the application

We pin an OAuth application to a single organization at consent time. `oauth_applications.organization_id` is a typed Postgres column, with the same RLS pattern as `api_keys.organization_id`. The API's auth middleware joins `oauth_access_tokens → oauth_applications` on every (cache-miss) validation and rejects tokens whose application has `organization_id IS NULL` — that catches the case where `/mcp/register` ran but the user never completed consent.

### Why we don't call Better Auth from the API

Better Auth's MCP plugin source does literally one thing in `withMcpAuth` / `getMcpSession`: a `findOne({ accessToken })` lookup on `oauth_access_tokens`. It does **not** check `accessTokenExpiresAt`, **not** check `oauth_applications.disabled`, **not** check scopes, **not** read the org binding. So routing through BA would buy us nothing on the API side — we'd still have to layer every real check on top of it.

The API's OAuth validator runs one SQL on cache miss:

```sql
SELECT t.user_id, t.client_id, t.scopes, t.access_token_expires_at,
       a.disabled         AS application_disabled,
       a.organization_id  AS organization_id
FROM   oauth_access_tokens t
JOIN   oauth_applications a ON a.client_id = t.client_id
WHERE  t.access_token = $1
LIMIT  1
```

…then rejects the row if any of: not found, expired, `applicationDisabled = true`, or `organizationId IS NULL`. That list IS the entire business logic. Tests in `packages/platform/oauth-token-auth/` pin each rule.

The platform package imports Drizzle table definitions from `@platform/db-postgres` and a `RedisClient` from `@platform/cache-redis`. No `better-auth` dependency anywhere on the API side. See [`api.md`](./api.md#authentication) for the caching shape and `AuthContext` type.

## Local development

No reverse proxy needed. Web at `localhost:3000`, API at `localhost:3001`, end-to-end OAuth works as-is:

- The browser navigation stays on `localhost:3000` through sign-in and consent — same origin as the dev session cookie.
- The MCP CLI (e.g. `npx @modelcontextprotocol/inspector`) talks to both origins via plain HTTP. CLIs don't enforce CORS.
- The API's `/.well-known/oauth-protected-resource` JSON points at `http://localhost:3000/api/auth` for dev (env-driven; falls back to the BA URL).

## Token revocation

Better Auth's MCP plugin has no `/revoke` endpoint. Revocation is implemented on the web app's settings UI as a server function: `DELETE FROM oauth_access_tokens WHERE client_id = ? AND user_id = ?`, and `oauth_applications.disabled = true` if no tokens remain for the application. The API's middleware honors the revocation immediately because validation hits the DB (modulo the 5-minute Redis cache; the cache invalidator in `@platform/oauth-token-auth` busts each removed token's entry at revoke time).
