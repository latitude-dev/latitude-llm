# TypeScript SDK

The official TypeScript SDK at `packages/sdk/typescript` (published as `@latitude-data/sdk`) is generated from `apps/api/openapi.json` by [Fern](https://buildwithfern.com/). The SDK is the recommended way for end users to call the public API — see [`api.md`](./api.md) for the underlying REST surface.

## Pipeline

```
apps/api/src/routes/*.ts
   │  (defineApiEndpoint + Zod schemas)
   ▼
apps/api/openapi.json              ← regenerated by `pnpm openapi:emit`
   │
   ▼
fern/generators.yml                ← input spec + generator config
   │
   ▼ (`pnpm generate:sdk`)
packages/sdk/typescript/src/       ← generated TS source (Fern overwrites)
   │
   ▼ (`pnpm --filter @latitude-data/sdk build`)
packages/sdk/typescript/dist/      ← ESM + CJS + .d.ts bundles (tsdown)
```

There is no separate IDL or hand-authored client — the SDK is derived end-to-end from the same Zod schemas that back the runtime HTTP routes. Schema `.describe(...)` text becomes JSDoc on every generated method and field.

## Layout

```
packages/sdk/typescript/
  package.json        ← hand-written shell (name, version, deps, build)
  tsdown.config.ts    ← bundler config
  tsconfig.json
  CHANGELOG.md        ← hand-maintained
  src/                ← Fern-owned; regenerated on every run
    index.ts
    Client.ts
    api/
    core/
    errors/
    environments.ts
  dist/               ← build output
```

`src/` is fully owned by Fern. Don't hand-edit anything there — the next `pnpm generate:sdk` will overwrite it. The hand-written shell lives one level up (`package.json`, `tsdown.config.ts`, etc.) and survives regeneration.

## Generating

```bash
pnpm generate:sdk
```

The repo-root script does three things:

1. `pnpm openapi:emit` — refreshes `apps/api/openapi.json` from the live route schemas.
2. `pnpm sdk:check` — runs `fern check` to surface OpenAPI validation issues.
3. `./fern/invoke.sh generate --group local --local --force` — invokes the Fern generator container (`fernapi/fern-typescript-node-sdk`) and writes the result into `packages/sdk/typescript/src/`.

Fern runs the generator inside Docker. The `fern/invoke.sh` wrapper handles two host-environment quirks before delegating to `fern`:

- Docker-socket discovery (OrbStack / Colima / newer Docker Desktop installs).
- Stripping `npm_config_*` env vars that pnpm exports — older `npm` (which the Fern CLI and the generator image both shell out to) spams unknown-config warnings otherwise.

Make sure Docker is running before invoking the script.

## Excluding endpoints from the SDK

Operational endpoints that aren't part of the product surface (health probes, OAuth discovery documents, internal callbacks) should not appear as SDK methods. Add `"x-fern-ignore": true` to the route's `createRoute` config — Fern's OpenAPI ingester honors that extension and skips the operation during generation:

```ts
const healthRoute = createRoute({
  method: "get",
  path: "/health",
  operationId: "health.get",
  tags: ["Health"],
  summary: "Health check",
  "x-fern-ignore": true,
  responses: { /* … */ },
})
```

These endpoints stay in `openapi.json` (consumers may still discover them at the spec level), but they're absent from `Client.ts`. They're also automatically absent from MCP because they bypass `defineApiEndpoint` (see [`mcp.md`](./mcp.md)).

## Versioning

The SDK is pre-1.0 and ships as `6.0.0-alpha.X`. Bump the alpha tail every time you publish:

- `6.0.0-alpha.1` → `6.0.0-alpha.2` for additive changes.
- A new alpha is fine for breaking changes too while we're in alpha — document them in the CHANGELOG.

The version lives in `packages/sdk/typescript/package.json`. Both internal consumers (`examples/sdks/typescript`, `packages/platform/latitude-api`) reference the SDK via `workspace:*`, so a version bump doesn't ripple inside the monorepo — it only affects the npm publish.

## Changelog

`packages/sdk/typescript/CHANGELOG.md` is **hand-maintained**, not auto-generated. The format follows [Keep a Changelog](https://keepachangelog.com/) and Semantic Versioning. Every pre-release bump gets its own section under the new version header with categorized bullets:

- **Added** — new resources, methods, types.
- **Changed** — modified shapes/signatures that callers may need to update.
- **Removed** — deleted types or methods (always paired with the replacement).
- **Breaking Changes** — sub-section of the above when consumers must update code.

Refresh the changelog in the same commit as the SDK regen, before tagging.

## Publishing

A dedicated workflow (`.github/workflows/publish-typescript-sdk.yml`) handles the npm publish. It triggers off git tags of the form `typescript-sdk-<version>`.

## Workspace consumers

Two in-repo packages already depend on the SDK via `workspace:*`:

- `examples/sdks/typescript` — runnable examples used in the public docs.
- `packages/platform/latitude-api` — internal wrapper used by other Latitude packages that need to call the API at runtime.

If you ship a breaking SDK change, check both call sites build cleanly before merging.
